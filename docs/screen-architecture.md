# 画面構成案（アルファMVP）

## 基本方針
- アルファでは「招待～共同スペース作成」と「共有買い物リスト体験」を最短経路で提供する。
- 画面数を絞り、モバイルファーストで操作の連続性とリアルタイム性を重視する。
- 将来の家事機能や通知設定はプレースホルダーとし、拡張時に情報設計を追加する。

## 必須画面一覧

### 1. ウェルカム／ログイン画面
#### パス
`/` （未ログイン時はログイン、ログイン済みならスペース作成へリダイレクト）
#### 目的
Supabase Auth を使い、既存アカウントでログイン or 新規登録。

#### 主要要素
ブランドロゴ、説明コピー、ログインフォーム、サインアップ誘導、（オプション）ソーシャルログインボタン。

#### フォーム
メール／パスワード入力。バリデーションはメール形式と8文字以上のパスワード。サインアップ時はパスワード再入力＋利用規約同意チェック。

#### サブミット／内部ロジック
Supabase Auth API を呼び出し、成功時に `profiles` を upsert してリダイレクト先を `couple_partners` の有無で分岐。

#### UI振る舞い
サブミット中はボタンをローディング表示。エラーはフォーム下にトースト表示。モバイルではキーボード出現時にロゴを縮小してフォームを見切れさせない。

### 2. カップルスペース作成画面
#### パス
`/couple/create`
#### 目的
最初のパートナーがスペースを作成し、基本情報を登録。

#### 主要要素
ページタイトル、説明テキスト、入力フォーム、利用規約リンク、プライマリーボタン。

#### フォーム
カップル名（任意・30文字以内）、タイムゾーン選択（必須）、招待メールアドレス（必須）、招待コード（英字4〜12文字で任意設定、未入力時は自動生成）。招待コードは半角アルファベットのみを許可し、リアルタイムバリデーションを表示。
- 初期表示でブラウザのタイムゾーンを自動入力し、取得できない場合は `UTC` をセットする。

#### サブミット／内部ロジック
送信時に `couples` を作成 → `couple_partners` に本人を登録 → `partner_invites` を作成。招待コード重複時は再生成してリトライ。完了後はダッシュボードへ遷移し、サクセストーストを表示。

#### UI振る舞い
未入力項目があるとボタンを無効化。タブレット幅以上では2カラムでフォームと説明を分割。招待コード欄に「自動生成」ボタンを配置して即時入力できるようにする。

### 3. 招待状況ダッシュボード
#### パス
`/couple/invitations`
#### 目的
参加状況の確認と招待再送。

#### 主要要素
ステータスカード（招待メール／状態／有効期限）、招待コード表示、コピーアイコン、再送・キャンセルボタン、参加済みメンバー表示。

#### フォーム
なし（インライン操作のみ）。

#### サブミット／内部ロジック
再送時は `partner_invites` の `status` と `expires_at` を更新し、メール再送 API を呼び出す。キャンセル時は `status` を `canceled` にし、一覧から除外。追加招待ボタンを押すと招待作成モーダルを開く。

#### UI振る舞い
状態に応じたラベルカラー（pending=黄、accepted=緑、expired=灰）。コピー操作はツールチップでフィードバック。リアルタイムに参加済み人数が更新される。

### 4. 招待受諾画面（公開リンク）
#### パス
`/invite/[code]`
#### 目的
招待されたパートナーがワンタップで参加。

#### 主要要素
招待の概要（招待者名・カップル名・有効期限）、参加ボタン、拒否ボタン、ログイン／新規登録案内。

#### フォーム
ログイン済みユーザーはフォーム不要。未ログインの場合はメールアドレスの再確認フォームとパスワード入力（または Supabase Magic Link）。

#### サブミット／内部ロジック
参加ボタン押下で `partner_invites` を `accepted` に更新し、`couple_partners` に参加者を作成。既に参加済みの場合はダッシュボードへリダイレクト。期限切れの場合はエラー表示と再招待依頼導線を提示。

#### UI振る舞い
有効期限が近づいている場合に警告バナー表示。成功時はスケルトンローディング後、買い物リスト一覧へスムーズスクロール付きで遷移。

### 5. 共有買い物リスト一覧
#### パス
`/lists`
#### 目的
カップルで共有するリストを切り替え。

#### 主要要素
リストカード（タイトル、最新更新者、未チェック件数）、フィルタタブ（アクティブ／アーカイブ）、新規作成ボタン、検索バー（任意）。

#### フォーム
新規リストモーダル（タイトル必須、初期カテゴリテンプレート選択、アーカイブ切り替え）。

#### サブミット／内部ロジック
作成時に `shopping_lists` を追加し、`is_active` を true で保存。アーカイブ操作は `is_active` を false に更新。リアルタイムサブスクリプションでリストの追加・更新を即反映。

#### UI振る舞い
カードタップで詳細画面へ。未チェック件数が0の場合はバッジを非表示。リストが無い場合はイラスト付き空状態を表示。

### 6. 買い物リスト詳細
#### パス
`/lists/[listId]`
#### 目的
アイテムの追加・編集・チェックを行うコア画面。

#### 主要要素
ヘッダー（リスト名、共有メンバーアイコン）、未チェック／チェック済みタブ、アイテムカード（名称・数量・追加者・最終更新）、インラインアクションメニュー、履歴へ切り替えるスイッチ。

#### フォーム
下部固定のアイテム追加フォーム（名称必須、数量／カテゴリ／メモ任意）。フォーム送信で `shopping_list_items` を作成し、`shopping_list_item_states` は初期値 false で保存。

#### サブミット／内部ロジック
チェック操作は `shopping_list_item_states` を更新（`is_checked` true/false、`checked_by`、`checked_at`）。アイテム削除は論理削除でなく実レコード削除。リアルタイム購読で他端末の変更を即反映。

#### UI振る舞い
チェック時に軽いアニメーションを付与。タブ切り替えでスクロール位置を保持。エラー時はインラインで表示し、失敗した操作をローカルでロールバック。
- リスト一覧 `/lists` からの遷移時は `NativeModal`（ボトムシート）で表示し、URL直接アクセス時のみフルページを表示。閉じると `/lists` に戻るため、モーダルは `router.back()` をトリガーする。

### 7. アイテム編集モーダル
#### パス
`/lists/[listId]`
#### 目的
リスト詳細でタップしたアイテムを編集。

#### 主要要素
アイテムの基本情報フォーム、最終更新者情報、削除リンク、閉じるボタン。

#### フォーム
名称（必須）、カテゴリ／数量／メモ（任意）。変更時はリアルタイムバリデーションで空文字を禁止。チェック状態の切り替えトグルも配置。

#### サブミット／内部ロジック
保存で `shopping_list_items` を更新。チェックトグル操作は `shopping_list_item_states` を更新。削除時は確認ダイアログ後にアイテムと状態テーブルを削除し、モーダルを閉じる。

#### UI振る舞い
モーダルはモバイルでは全画面表示、デスクトップではセンタリング。保存完了でスナックバーを表示し、背景リストを自動更新。

### 8. スペース設定画面
#### パス
`/settings/couple`
#### 目的
カップル情報・メンバー管理の変更。

#### 主要要素
カップル名編集カード、タイムゾーンセレクター、メンバー一覧（役割・参加日）、招待リンク再発行ボタン、スペース退会／削除セクション。

#### フォーム
カップル名（任意）、タイムゾーン（必須）。メンバーの表示名編集フォームを配置するかは後続検討。

#### サブミット／内部ロジック
情報更新時に `couples` を更新。メンバーの退会操作は `couple_partners` を削除し、残された側に通知。スペース削除は `couples` と関連データをトランザクションで削除。

#### UI振る舞い
保存成功時にトースト表示。危険操作（退会・削除）は赤ボタン＋確認ダイアログ必須。モバイルではアコーディオンでセクションを折り畳む。

### 9. プロフィール設定
#### パス
`/settings/profile`
#### 目的
表示名・性別・アイコンの編集。

#### 主要要素
アバターアップローダー、表示名入力、性別入力欄、保存ボタン、アカウント設定リンク。

#### フォーム
表示名（必須・30文字以内）、性別（任意入力テキスト）、アバター画像（最大5MB・正方形推奨）。

#### サブミット／内部ロジック
画像アップロード後に Supabase Storage の URL を取得し、`profiles` を更新。表示名は他ユーザーと重複可。更新完了でアプリ全体の表示名を即時反映。

#### UI振る舞い
フィールド変更時にリアルタイムプレビュー。保存中はボタンをローディング表示。入力未変更の場合はボタンを無効化。

## 画面遷移概略
1. **ウェルカム** → (ログイン/サインアップ) → **カップルスペース作成**（初回） → **招待状況ダッシュボード**。
2. 招待メール → **招待受諾画面** → ログイン／アカウント作成 → **買い物リスト一覧**。
3. 一覧 → **買い物リスト詳細** → アイテムタップ → **アイテム編集モーダル**。
4. ナビゲーション（ヘッダー/ボトム）から **スペース設定** や **プロフィール設定** へ遷移。

## 未来拡張のプレースホルダー
- **家事ボード画面**: `chores` 追加後に導入。買い物リスト詳細と構造を揃えると導線が理解しやすい。
- **通知管理画面**: `notification_preferences` を導入するタイミングで、通知チャネル・頻度の設定 UI を追加。
- **ダイジェスト／アクティビティ画面**: 完了履歴や賞賛（将来機能）を振り返るタイムラインとして検討。

## 実装メモ
- 主要画面（5,6）はモバイル優先で縦スクロール＋スワイプ操作を検討。
- Realtime サブスクリプションはリスト詳細とアイテム状態にのみ適用し、他画面はポーリング or ページ遷移時に再フェッチ。
- スペース未参加状態のユーザーには招待受諾専用 UI を表示し、買い物リスト関連コンポーネントをマウントしないようガードする。
- カレンダー画面は `ProtectedLayout` のブランドヘッダーのみを使い、画面内部に重複するタイトルや説明テキストを置かない（未参加状態のみ説明カードを表示）。
- カレンダーの日別詳細は `DayDetailModal` 一枚の `NativeModal` で完結させ、イベントカード一覧→イベント詳細閲覧→編集フォームまでを同じボトムシート内で遷移させる。`EventDetailContent` をモーダル内に埋め込み、戻るボタンで一覧へ戻れるようにしつつ、閉じるボタンはシート全体を閉じる。これによりイベント選択時に別モーダルへ切り替わる違和感や `toLocaleDateString` エラーの再発を防ぎ、編集後も同じ日付の他イベントを続けて操作できる。
- データフェッチが必要な領域は `page.tsx` 内に Suspense 境界を持たせ、`loading.tsx` は原則配置せず Suspense の `fallback` で共有スケルトンを表示する。静的テキストやヘッダーは SSR された `layout`/`page` で一度だけ定義し、二重実装を避ける。
- フォーム送信は共通のバリデーションレイヤー（React Hook Form + Zod想定）を使用し、ローディング・成功・失敗状態のトーストを統一する。

## 共通UIガイドライン
- **ローディング状態**: フォーム送信時はサブミットボタンを `disabled` にし、スピナーを表示。非同期処理が完了するまで多重送信を防ぐ。
- **入力バリデーション**: 全フォームでリアルタイムフィードバックを行う。エラーはフィールド直下にメッセージを表示し、サブミット時はトーストでも通知する。
- **トースト通知**: 成功・失敗・情報の3種を用意。成功はグリーン、失敗はレッド、情報はプライマリーを基調とし、3秒後にフェードアウト。
- **ダイアログ**: 危険操作（削除・退会）は必ず確認ダイアログを挟む。ダイアログのデフォルトフォーカスはキャンセルボタン。
- **リアルタイム更新**: リスト詳細などのリアルタイム対象画面では、更新内容を軽いアニメーションで強調し、同時操作時は最新データを優先してマージする。
- **空状態の表現**: データがない場合は説明テキスト＋アイコン／イラストを表示し、次の操作（例: アイテム追加）の導線を強調する。
- **レスポンシブ**: モバイルを基準としつつ、タブレット以上では2カラムレイアウトやモーダルの中央表示で余白を活かす。
- **モーダル**: `NativeModal` は背景に半透明のオーバーレイを敷き、モバイルではフッターからせり上がるシート（上部のみ余白＋大きめの角丸）として前画面の上に被せる。内容が少なくても `min-h` を `calc(100vh - 3rem)` に設定し、常に画面の大部分を覆う。`sm` 以上は中央寄せ＋`max-width`/`max-height` を適用し、内容は `overflow-y-auto` で長文でもスクロールできるようにする。ヘッダーやボトムナビより高い `z-index` 変数を割り当て、常に最前面で表示されるようにする。開閉時はモバイルで 300ms のスライド（下→上で表示／上→下で非表示）＋フェード、デスクトップで 200ms のフェード＋わずかなバウンスを付け、ネイティブアプリのモーダルスタックと同じフィードバックを再現する。親コンポーネントは `isOpen` プロップを切り替え、閉じアニメーション完了まではアンマウントせず `NativeModal` 内部または親側のタイマーで待機する。
- **アクセシビリティ**: 主要ボタンに加えキーボード操作（Tab/Enter/Space）とスクリーンリーダー向けラベルを付与。カラーコントラストはW3C AA準拠。
- **共通コンポーネント**: ボタン、入力フィールド、モーダル、トーストは `src/common/ui/` のデザインシステムを利用し、状態遷移アニメーションや余白を統一する。
- **フィードバック速度**: 操作後300ms以内に視覚的なフィードバック（ローディング／成功アニメーション）を返し、ネットワーク待機時は骨組み表示（スケルトン）を活用する。
- **スケルトン方針**: タイトルや導線テキストなど静的コンテンツはSSRで即座に描画し、ローディング中も実テキストを保持する。スケルトンはAPI応答に依存するリストやカード領域に限定し、画面骨子が透明化しないようにする。
